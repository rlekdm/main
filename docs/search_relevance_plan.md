# 검색 결과 정합성 판별 모델 진행 계획

이 문서는 크롤링 결과에서 `keyword`를 표준 상품명으로 사용하기 위해, 검색 결과가 실제 해당 상품군에 맞는지 판별하는 다음 개발 방향을 정리합니다.

## 목표

- `keyword_list.csv`의 `keyword`를 표준 상품명(`canonical_name`)으로 관리합니다.
- 크롤링된 원본 상품명(`name`)이 해당 표준 상품명에 맞는 결과인지 판별합니다.
- 상품별 최저가, 평균가, 가격 변동을 `canonical_name` 기준으로 안정적으로 집계할 수 있도록 합니다.

## 현재 진행 상황

- `code/backend/src/main/python/crawling/keyword_list.csv`를 표준 상품명 후보 목록으로 사용하고 있습니다.
- `crawling_20260429.py`는 최근 14일 데이터를 수집합니다.
- 키워드별 개별 CSV가 아니라 `통합조회_전체_YYYYMMDD_HHMM.csv` 하나만 출력합니다.
- 같은 상품이 여러 키워드에 걸리면 `platform + pid` 기준으로 중복 제거합니다.
- 중복 상품의 `matched_keywords` 중 더 구체적인 긴 키워드를 `canonical_name`으로 선택합니다.
- `+`, `plus`, `플러스`, `pro`, `max`, `ultra` 같은 표기 차이를 정규화합니다.
- `crawling_20260429_no_filter.py`를 만들어 필터 미적용 결과와 비교할 수 있게 했습니다.

## 확인된 문제

- 번개장터 API는 검색어와 느슨하게 매칭된 후보를 많이 반환합니다.
  - 예: `갤럭시 s26` 검색 결과에 `갤럭시s23FE 256GB`, `갤럭시S24 256G` 등이 포함됩니다.
- 중고나라는 웹 검색 결과를 파싱하기 때문에 상대적으로 검색어 정합성이 높습니다.
- `아이폰 17e`처럼 숫자와 짧은 알파벳 접미사가 붙은 모델명은 오탐에 취약합니다.
  - `17e`가 `17`과 `e`로 나뉘면 `case`, `MagSafe`, `Sweet` 같은 단어의 `e` 때문에 잘못 매칭될 수 있습니다.
- 케이스, 필름, 맥세이프 같은 액세서리 상품이 본체 가격 통계에 섞일 수 있습니다.
- no-filter 결과는 수량이 늘지만 상품명 정합성이 낮아져 가격 통계 품질을 떨어뜨릴 수 있습니다.

## 비교 결과 요약

기준 파일:

- 필터 적용: `통합조회_전체_20260507_0522.csv`
- 필터 없음: `통합조회_전체_no_filter_20260507_0536.csv`

집계:

- 필터 적용 결과: 5,515건
- 필터 없음 결과: 8,078건
- 차이: 2,563건

플랫폼별:

- 번개장터: 3,606건 -> 6,162건
- 중고나라: 1,909건 -> 1,916건

해석:

- 필터 유무 차이는 대부분 번개장터에서 발생합니다.
- 중고나라는 검색 결과 자체가 비교적 정제되어 있어 필터 전후 차이가 작습니다.
- 번개장터는 API가 검색 후보군에 가까운 넓은 결과를 반환하므로 별도 정합성 판별이 중요합니다.

## 다음 개발 방향

### 1. 라벨 데이터셋 만들기

검색 결과 정합성 판별 모델을 만들기 위해 `keyword`, `name`, `platform`, `price`, `label` 형태의 학습 데이터를 만듭니다.

권장 라벨 기준:

- `label = 1`: 해당 `keyword`의 실제 상품군에 맞는 결과
- `label = 0`: 다른 모델, 액세서리, 호환 상품, 케이스, 필름, 단순 나열 상품

초기 라벨링 전략:

- 중고나라 필터 적용 결과를 positive 후보로 사용합니다.
- 번개장터 no-filter 결과 중 현재 필터에서 제거된 데이터를 negative 후보로 사용합니다.
- 애매한 케이스는 수동 검토용 CSV로 분리합니다.

### 2. 특징 추출

우선 딥러닝보다 규칙 기반 특징과 간단한 분류 모델을 사용합니다.

후보 특징:

- `keyword` 토큰이 `name`에 모두 포함되는지
- 모델명 토큰 일치 여부: `s25`, `s26`, `17e`
- 숫자 토큰 일치 여부
- 접미어 일치 여부: `플러스`, `프로`, `맥스`, `울트라`
- 액세서리 단어 포함 여부: `케이스`, `필름`, `강화유리`, `맥세이프`, `magsafe`, `충전기`
- 여러 모델 나열 여부: `13,14,15`, `/`, `or`, `호환`, `전기종`, `공용`
- 가격 범위
- 제목 길이
- 플랫폼

### 3. 1차 모델

처음에는 설명 가능한 모델을 사용합니다.

- `scikit-learn` Logistic Regression
- RandomForest
- LightGBM 또는 XGBoost

모델 목적:

- `keyword`와 `name`의 정합성 점수 산출
- 일정 점수 이상만 DB 저장 대상으로 사용
- 점수가 낮거나 애매한 결과는 검토 대상으로 분리

### 4. 고도화 후보

데이터가 충분히 쌓이면 아래 방향을 검토합니다.

- `rapidfuzz`로 문자열 유사도 특징 추가
- `sentence-transformers` 또는 한국어 SBERT 계열 임베딩 활용
- Learning to Rank 방식으로 검색 결과 정렬 고도화
- Entity Resolution 방식으로 동일 표준 상품명 매칭 개선

## DB 저장 방향

크롤링 결과를 DB에 넣을 때는 원본 상품명과 표준 상품명을 분리합니다.

- `canonical_name`: 표준 상품명, 가격 집계 기준
- `title`: 플랫폼 원본 상품명
- `matched_keywords`: 같은 상품에 매칭된 표준 상품명 후보 목록
- `platform_id + original_id`: 중복 저장 방지 기준

가격 통계는 `canonical_name` 또는 향후 `product_id` 기준으로 집계합니다.

예시:

```sql
SELECT
  canonical_name,
  MIN(current_price) AS lowest_price,
  AVG(current_price) AS average_price
FROM Items
GROUP BY canonical_name;
```

## 우선순위

1. `17e`, `s25`, `s26`처럼 숫자+문자 모델명을 하나의 토큰으로 처리합니다.
2. 액세서리 제외 후보 단어를 정리합니다.
3. 필터 적용 결과와 no-filter 결과를 이용해 라벨 후보 CSV를 생성합니다.
4. 라벨 후보를 일부 수동 검수해 학습/검증 데이터를 만듭니다.
5. `scikit-learn` 기반 1차 정합성 분류 모델을 구현합니다.
6. 모델 점수와 기존 규칙 필터를 함께 사용해 DB 저장 대상을 결정합니다.
